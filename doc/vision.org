#+title: org-slipbox Vision

* North Star

`org-slipbox` is a local-first Org slip-box for serious personal knowledge
work.

Its job is to help a user think with a corpus of notes: find stable facts,
surface meaningful relations, expose weak or stale areas, preserve lines of
inquiry, and support deliberate maintenance. The system should remain plain
Org at rest, fast in use, and explicit about every durable record it creates.

* Product Thesis

- Org files are the source of truth.
- The SQLite database is a derived, rebuildable index.
- Expensive parsing, indexing, ranking, query execution, and file mutation
  belong in Rust.
- Emacs provides the rich interactive cockpit.
- The CLI provides the scriptable everyday and workbench surface.
- Public concepts follow the compact model in [[file:model.org][doc/model.org]].
- Growth should make the system easier to reason about, package, and operate.

* Thinking Surface

The system should answer more than "what links here?"

It should help answer:

- What nearby material is relevant but weakly integrated?
- What other lines of thought does this note open?
- Which refs, tasks, dates, or links recur across distant regions?
- Which claims, plans, or examples are in tension?
- Which material has gone dormant but is now useful?

This requires explainable evidence, factual comparison, explicit trails,
stable saved artifacts, and review loops that keep maintenance work separate
from note identity.

* One Model, Multiple Surfaces

The Emacs package, CLI, daemon JSON-RPC surface, and future MCP surface should
share one model:

- Notes and Relations come from Org.
- Explorations derive meaning from indexed evidence.
- Reviews record operational findings and user decisions.
- Assets describe reusable workbench behavior as data.
- System surfaces report daemon, index, file, diagnostic, benchmark, and
  compatibility state.

Each surface may present the model differently, but semantic ownership should
stay the same. The dedicated Emacs buffer is the richest interactive cockpit.
The CLI is the task-shaped headless surface. Future MCP work should map this
same model to resources and tools rather than inventing another ontology.

* Architecture

- Rust owns parse/index/query/write semantics.
- Emacs Lisp owns commands, buffers, session state, and presentation.
- Daemon-backed automation goes through `slipbox serve`.
- File-level updates stay incremental and explicit.
- Public durable records live outside the derived SQLite index.
- Persistent UI paths stay cheap; explicit cockpit and headless commands own
  richer discovery.
- Central files should be split along product-domain seams as the model grows.

* Product Standards

- Claimed workflows must work end to end.
- Surprising results need concrete evidence.
- Writes need explicit affected-file reporting and read-your-writes behavior.
- Review state stays distinct from notes and exploration artifacts.
- Declarative assets stay data, with validation and compatibility metadata.
- Public JSON contracts evolve deliberately and additively where possible.
- Performance claims require benchmark gates over real fixtures.

* Measure Of Success

`org-slipbox` is succeeding when:

- ordinary note work remains fast and predictable at large corpus sizes
- users can discover structure they did not manually encode ahead of time
- reviews and diagnostics make maintenance pressure visible
- Emacs and CLI feel like two surfaces over one slip-box
- package and release paths are boring
- future integration surfaces can reuse the model without rescue refactors
