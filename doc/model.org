#+title: org-slipbox Public Model

* Purpose

This document defines the public model used by the documentation, CLI
taxonomy, durable JSON contracts, and future MCP mapping.

`org-slipbox` has six public buckets:

- Notes
- Relations
- Explorations
- Reviews
- Assets
- System

Everything shipped by the project should belong to exactly one of these
buckets. Smaller concepts are named inside their owning bucket rather than
promoted into separate public domains.

* Model Rules

- Org files are the source of truth for notes and note relations.
- The SQLite database is a derived index. It may be rebuilt and remains an
  internal data model.
- Durable side stores live beside the database and remain separate from notes,
  search, refs, and graph identity.
- The daemon owns parsing, indexing, querying, ranking, and file mutation.
- Emacs and the CLI are two front-ends over the same daemon-owned model.
- JSON-RPC is the transport boundary; public commands and durable records stay
  task-shaped.

Command ownership and record ownership are separate. For example,
`workflow run --save-review` and `routine run` remain Asset commands, but the
review records they create belong to Reviews. `node search` remains part of
the Notes command family, while top-level `search occurrences` belongs to
Relations.

** Duplicate Affordances

These decisions feed later alias/deprecation work. This section records the
taxonomy decision only; it does not change command behavior.

| Affordance | Bucket | Decision |
|------------+--------+----------|
| `ref show` and `ref resolve` | Relations | Prefer `ref resolve` as the canonical verb because the command resolves a ref to a note. Keep `ref show` as a compatibility alias for now; consider a deprecation notice only after command-help and contract policy work make warnings consistent. |
| `note create` and `capture node` | Notes | Prefer `note create` as the canonical everyday command for creating a file note. Keep `capture node` as the capture-namespace spelling for users working from capture flows. The implementation may stay shared. |
| `node backlinks` / `node forward-links` returning relation evidence | Notes / Relations | Keep command ownership under Notes because the user starts from an exact note target. The returned backlink/forward-link evidence belongs to Relations. |
| `node metadata show` and top-level `node alias/ref/tag` mutators | Notes | Keep. `node metadata show` is the read surface; `node alias`, `node ref`, and `node tag` are focused write shortcuts over the same note metadata. |
| `workflow show --spec` and `pack validate` local inspection | Assets | Keep. Both are local asset-inspection paths, but one inspects a workflow spec and the other validates a pack manifest. Help should make the local-vs-daemon distinction clear. |
| `artifact run` and `explore` / `compare` live commands | Explorations | Keep. `artifact run` executes a saved exploration artifact; `explore` and `compare` run live queries and can optionally save new artifacts. |
| `audit ... --save-review`, `workflow run --save-review`, and `routine run` review output | Reviews / Assets | Keep the command owners as-is. Assets own workflow/routine execution; Reviews own the review records produced by save-review paths. |
| `diagnose file` and `file list/search` | System | Keep. File commands expose indexed file inventory; diagnostics explain eligibility and index drift. |

* Persistence Boundaries

Notes and Relations are persisted in Org files. The SQLite database accelerates
queries over them and may be rebuilt.

Explorations, Reviews, and imported Assets use explicit durable JSON side
stores next to the database:

- exploration artifacts: `<database-name>.exploration-artifacts/v1/`
- review runs: `<database-name>.review-runs/v1/`
- imported workbench packs: `<database-name>.workbench-packs/v1/`

Those side stores survive index rebuilds while staying outside note identity,
ref identity, search hits, and graph edges.

* Compatibility Boundary

Public compatibility covers:

- CLI `--json` and `--jsonl` envelopes
- JSON-RPC method params and results
- durable exploration artifacts
- durable review runs
- workflow specs and workbench packs with compatibility version `1`
- review routines and report profiles carried by packs
- structural write reports, diagnostics, link rewrite records, and remediation
  apply records

Compatibility does not cover derived SQLite tables, internal ranking details,
private helper modules, or benchmark harness implementation internals.

The authoritative policy lives in [[file:compatibility.org][doc/compatibility.org]].
