#+title: org-slipbox Roadmap

* Purpose

This roadmap maps release bands to the vision and milestones. It intentionally
avoids time promises.

* 0.2.x

Theme: settle the first release and lay the cockpit foundation.

Focus:

- strengthen the dedicated buffer as an exploratory surface
- improve explanatory and navigational affordances
- deepen diagnostics, docs, and operator confidence
- preserve the thin-client architecture under this new UI pressure

Watchpoints:

- do not let buffer richness reintroduce expensive Emacs-side hot paths
- do not let exploratory features become ad hoc grep or shell glue

* 0.4.x

Theme: real conversational exploration.

Focus:

- multiple relatedness lenses
- explicit pivots, comparisons, and trails
- surfacing bridges, dormant threads, and unresolved material
- richer index-backed reasoning over tasks, refs, and time
- only the minimum shared exploratory semantics needed to make the cockpit
  real, not the broader programmable workbench surface
- keep saved views, broader CLI/headless flows, and extension/platform work
  deferred until the cockpit model has genuinely settled

Watchpoints:

- keep every surprising result explainable
- do not replace structural discovery with vague ranking theatrics
- do not let transport, CLI, or extension concerns freeze the model too early

* 0.5.x

Theme: settle the cockpit model.

Focus:

- coherent lens-local rendering and explanation presentation
- richer task, ref, and time reasoning inside dedicated exploration
- factual comparison of overlap, divergence, ambiguity, and tension
- clearer trail replay, detached branching, and cockpit legibility
- benchmark and regression coverage for real dedicated exploration paths
- keep saved views, broader headless workflows, and public workbench
  extraction explicitly deferred

Watchpoints:

- do not mistake cockpit settlement for programmable-workbench readiness
- do not let docs imply saved-view or headless maturity that does not yet
  exist
- do not let “polish” work dissolve the hard persistent-versus-dedicated
  cost split

* 0.6.x

Theme: first workbench foundation.

Focus:

- durable exploration artifacts for saved views, trails, and comparisons
- Rust-owned artifact semantics, persistence, and execution
- dedicated-cockpit save, load, and replay over those artifacts
- first narrow task-shaped machine operations around those artifacts:
  save, inspect, list, execute, and delete
- keep broad CLI families, extension APIs, and agent adapters deferred

Watchpoints:

- do not let saved artifacts pollute public note semantics
- do not reconstruct artifact semantics in Emacs Lisp
- do not mirror raw transport methods when the public unit is a meaningful
  exploratory operation
- do not let workbench extraction weaken the persistent-versus-dedicated cost
  split

* 0.7.x

Theme: first usable headless workbench.

Focus:

- a typed Rust daemon client over the canonical JSON-RPC stdio boundary
- shared CLI runtime, connection, and JSON-first output scaffolding
- task-shaped live headless operations for resolve, explore, and compare
- durable artifact lifecycle commands for list, show, run, delete, import,
  export, and save from live exploration/comparison flows
- end-to-end CLI coverage that proves the headless surface is real rather than
  transport-shaped aspiration
- documentation that treats this as the first usable headless workbench rather
  than as broad platform completion

Watchpoints:

- do not bypass `slipbox serve` through in-process shortcuts
- do not turn the CLI into a thin wrapper over every RPC method
- do not invent a second ontology for headless work that drifts from cockpit
  semantics
- do not let JSON-first machine use turn into “just use raw RPCs” instead of
  documenting the actual CLI house
- do not let extension APIs, MCP, or agent-adapter ambitions leak into this
  milestone

* 0.8.x

Theme: broader research workbench.

Focus:

- named workflows for common research routines over the first usable headless
  workbench and the durable artifacts from the settled cockpit model
- corpus-health audit surfaces over the canonical daemon boundary
- bounded workflow discovery from configured directories
- workflow and audit report outputs for JSON and line-oriented machine use
- broader headless composition without widening into transport-shaped CLI
  sprawl
- larger benchmark corpora and stricter scale guarantees for the larger
  workbench surface

Watchpoints:

- do not mirror raw RPCs when the real unit of value is a named workflow or
  audit operation
- treat workflow discovery as declarative extension, not as a plugin runtime
- document JSON and JSONL workflow/audit output through the actual CLI
  contract rather than through raw RPC examples
- keep MCP, agent-adapter, and broader extension ambitions out of this band
- keep the store, write engine, and protocol growth modular
- do not let “broader workbench” turn into a fuzzy platform bucket

* 0.9.x

Theme: operational research workbench.

Focus:

- durable review runs for recurring named-workflow and corpus-audit loops
- explicit review status so repeated runs can be triaged without becoming a
  task-manager clone
- review diffs that show what changed since an earlier run
- safe remediation previews for audit findings and workflow outcomes before
  any write occurs
- a clear boundary between review records, ordinary notes, and saved
  exploration artifacts
- operational loops over the composed `0.8.x` workbench without widening into
  raw-RPC sprawl or broad automation

Watchpoints:

- do not make review records part of note identity, search identity, ref
  identity, or saved exploration-artifact identity
- do not imply that review status is a general task-management system
- do not execute broad structural writes when the right product surface is a
  preview and an explicit later decision
- do not let remediation previews bypass the Rust-owned write path
- keep extension APIs, plugin-runtime work, MCP, and agent-adapter ambitions
  out of this band
- keep the one exploratory model intact across cockpit, daemon, CLI,
  workflows, audits, and reviews

* 1.0.x

Theme: dependable system.

Focus:

- stable public surfaces
- explicit upgrade and compatibility expectations
- mature manual and extension guidance
- routine packaging and release engineering

Watchpoints:

- do not rush a `1.0.0` label until the exploratory model, not just the
  replacement surface, has proven stable

* Release Discipline

Every release band should satisfy the following:

- architecture first, then feature count
- measured performance, not inferred performance
- clean packaging and installation
- deliberate documentation of any meaningful divergences
- no work that makes future rescue refactors more likely
