#+title: org-slipbox Roadmap

* Purpose

This roadmap maps near-term release bands to the public model in
[[file:model.org][doc/model.org]]. It covers direction, not dates.

* 0.13.x: Consolidation

Theme: make the project smaller and easier to reason about before external
adapter work.

Focus:

- compact public ontology and command taxonomy
- README, roadmap, milestone, compatibility, and readiness docs cleaned up
  around the shared model
- `slipbox-core`, CLI, server handlers, benchmark harness, and test helpers
  split along product-domain seams
- no-behavior-change contract matrix kept green through refactors
- deliberate decisions for duplicate command affordances
- manpage-grade command help aligned with the taxonomy
- consolidation scorecard with line-count and term-count evidence

Exit signal:

- largest files are visibly smaller or mechanically split
- docs read as durable guidance rather than release notes
- CLI/RPC/JSON behavior is preserved except for reviewed aliases or
  deprecations
- public concepts map cleanly to Notes, Relations, Explorations, Reviews,
  Assets, and System

* 0.14.x: Read-Only MCP

Theme: expose the cleaned model through read-only resources and tools.

Focus:

- resource mapping for notes, files, refs, tags, artifacts, reviews,
  workflows, routines, packs, selected graph views, and selected exploration
  views
- read-only tools for lookup, search, backlinks, refs, tags, agenda, graph
  export, exploration, comparison, artifact execution, audit inspection,
  review inspection, workflow/routine catalog inspection, and diagnostics
- permission and scope model for local slipbox roots
- contract tests proving adapter output agrees with existing CLI/RPC semantics
- docs showing the mapping from the compact model to resources and tools

Exit signal:

- read-only clients can inspect and explore the corpus without mutating Org
  files or durable side stores
- the adapter reuses daemon-owned operations
- adapter naming follows the compact public ontology

* 0.15.x: Write-Capable MCP

Theme: add selected write-capable tools after read-only semantics have proven
stable.

Focus:

- permission-gated note creation, capture, metadata, identity, and daily-note
  operations
- permission-gated structural edits with existing `StructuralWriteReport`
  records
- preview/apply tools for supported remediation and link rewrite operations
- explicit affected-file, confirmation, and read-your-writes behavior
- tests that compare adapter write results with CLI/RPC write contracts

Exit signal:

- write-capable tools mutate only through existing Rust-owned daemon paths
- every write returns the same task evidence as the CLI/RPC surface
- preview/apply flows remain bounded and auditable

* 1.0.x: Stability Line

Theme: make the public surface dependable enough for ordinary users,
packagers, scripts, and integration authors.

Focus:

- stable public JSON and durable record compatibility policy
- documented upgrade and rebuild behavior
- reliable release binaries and source builds
- benchmark gates that cover everyday, exploratory, review, asset, structural,
  and adapter paths
- cleaned module boundaries that allow ordinary maintenance without rescue
  refactors
- complete user-facing documentation and command help

Exit signal:

- one full release cycle passes with the public contracts holding steady
- bundled and system-SQLite build paths are verified in release automation
- compatibility-sensitive changes have explicit migration guidance
- the project can evolve through modules instead of monolith pressure
