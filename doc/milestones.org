#+title: org-slipbox Milestones

* Purpose

This document names durable capability milestones rather than release history.

The public model is defined in [[file:model.org][doc/model.org]].

* Milestone 1: Stable Replacement Foundation

Purpose: make Org notes, IDs, refs, capture, backlinks, graph export, dailies,
and agenda work through a clean Emacs package plus Rust daemon.

Outcomes:

- Org files stay authoritative.
- The daemon and Emacs package are separable.
- Indexing, search, backlinks, refs, graph, capture, dailies, and agenda are
  covered.
- Packaging and installation are routine.
- Benchmark gates exist for core paths.

Exit criteria:

- ordinary note creation, lookup, linking, ref lookup, and context inspection
  are reliable
- index state can be rebuilt from Org files
- interactive hot paths avoid corpus-wide Emacs Lisp work

* Milestone 2: Exploratory Cockpit

Purpose: turn the current-node buffer into a real thinking surface.

Outcomes:

- cheap persistent context buffer
- richer dedicated cockpit
- declared lenses over structure, refs, time, tasks, bridges, dormant material,
  unresolved material, and local graph evidence
- factual comparison and explicit trails
- explanation blocks for non-obvious results

Exit criteria:

- users can pivot, compare, branch, and return to lines of inquiry
- exploratory results are explainable and index-backed
- persistent redisplay remains cheap

* Milestone 3: Durable Workbench

Purpose: make exploration, review, and reusable workbench behavior durable and
scriptable through the daemon boundary.

Outcomes:

- saved exploration artifacts for lenses, comparisons, trails, and detached
  trail slices
- headless resolve, explore, compare, and artifact lifecycle commands
- named workflows and bounded workflow discovery
- corpus-health audits and report output surfaces
- durable review runs with status, diffs, and remediation previews
- declarative workflow specs, review routines, report profiles, and workbench
  packs
- first-class everyday CLI operations across Notes, Relations, Explorations,
  Reviews, Assets, and System

Exit criteria:

- saved artifacts can be persisted, replayed, and executed
- reviews are durable records separate from notes and artifacts
- assets are portable JSON data with validation and compatibility metadata
- CLI commands remain task-shaped and daemon-backed
- contract tests cover machine-readable JSON envelopes

* Milestone 4: Safe Structural Operations

Purpose: expose higher-risk writes through explicit daemon-owned contracts.

Outcomes:

- refile, extract, promote, and demote operations with structural write reports
- link rewrite preview/apply records
- selected remediation apply records
- diagnostics for file, node, and index state
- rebuild-survival coverage for durable side stores

Exit criteria:

- every structural write reports affected files and refreshed-index state
- preview/apply operations require explicit intent
- follow-up reads observe the written state
- durable side stores survive index rebuilds without polluting note surfaces

* Milestone 5: Consolidated Public Surface

Purpose: make the product model, command taxonomy, docs, and code structure
small enough to support future adapters without carrying release-history
weight.

Outcomes:

- six-bucket public ontology
- compact README and strategic docs
- compatibility policy in one authoritative place
- CLI taxonomy and help aligned with the model
- core, CLI, server, benchmark, and test code split by product domain
- duplicate command affordances resolved deliberately

Exit criteria:

- public names map cleanly to Notes, Relations, Explorations, Reviews, Assets,
  and System
- largest omnibus files are split behind stable reexports or facades
- contract matrix proves behavior preservation through refactors
- consolidation scorecard shows line-count, doc-count, and term-count evidence

* Milestone 6: External Integration Surface

Purpose: map the cleaned model to external tools without changing the source
of truth or duplicating semantics.

Outcomes:

- read-only resources and tools over existing daemon semantics
- optional write-capable tools for selected safe operations
- permission model for read and write operations
- adapter tests matched against CLI/RPC contracts

Exit criteria:

- external clients can inspect the corpus through the same model as Emacs and
  CLI
- write tools use existing daemon-owned mutations and task reports
- adapter naming follows the compact ontology

* Milestone 7: Dependable System

Purpose: reach a stable, boring public surface.

Outcomes:

- stable compatibility and deprecation policy
- reliable release engineering
- verified bundled and system-SQLite build paths
- documented upgrade and rebuild behavior
- benchmark profiles that cover the public product surface

Exit criteria:

- public JSON and durable record formats hold steady across a release cycle
- packaging and installation are predictable
- users and scripts can rely on documented contracts
- new work lands through focused modules rather than monolith growth
