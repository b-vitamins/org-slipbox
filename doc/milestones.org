#+title: org-slipbox Milestones

* Purpose

This document translates the product vision into concrete software milestones.
These are capability milestones, not date commitments.

* Milestone 1: Stable Replacement Foundation

The first milestone is already achieved.

Outcomes:

- `org-slipbox` is a credible day-to-day replacement for `org-roam`
- the daemon and Emacs package are cleanly separable
- manual-backed workflows are covered
- correctness, scale, and packaging are proven rather than assumed

Exit criteria:

- the core node, capture, backlinks, refs, dailies, export, graph, and agenda
  surfaces are present
- packaging and installation are routine
- benchmark gates exist

* Milestone 2: Cockpit Foundation

Turn the dedicated context buffer into a first-class exploratory surface rather
than a status panel.

Outcomes:

- richer indexed section families
- explicit dedicated-buffer navigation and session state
- explicit pivot controls between relatedness lenses
- stable explanation of why sections and nodes are shown
- cheap persistent buffer, deeper dedicated buffer

Exit criteria:

- a user can navigate through the dedicated buffer rather than merely inspect
  it
- the persistent buffer remains cheap by design
- new buffer power does not move hot work back into Emacs Lisp

* Milestone 3: Conversational Exploration

Let the system surface lines of thought rather than only direct graph
adjacencies.

Outcomes:

- bridge detection between distant regions
- dormant or weakly connected material surfacing around the current context
- explicit comparison and trail management
- better support for ambiguity, tension, and unresolved work across refs,
  planning dates, and task states
- the exploratory model becomes durable enough to extract beyond the cockpit

Exit criteria:

- the system can show more than backlinks and nearest neighbors
- exploratory results remain explainable and index-backed
- users can revisit and branch explicit lines of inquiry in the cockpit,
  while durable saved views remain later workbench work
- cockpit behavior rests on explicit shared semantics rather than buffer-local
  improvisation
- the cockpit is documented and tested as a coherent exploratory surface, not
  a bag of sections

* Milestone 4: Programmable Research Workbench

Expose the engine and exploratory model as a serious programmable environment.

Outcomes:

- durable exploration artifacts for saved views, trails, and comparisons
- query surfaces and first task-shaped public operations around those artifacts
  such as save, inspect, list, execute, and delete
- a first usable headless workbench around resolve, live explore, compare, and
  artifact lifecycle over the canonical daemon boundary
- named workflows for common research routines over that settled headless
  model
- corpus-health audit surfaces that expose maintenance pressure and weak
  integration as first-class workbench operations
- bounded workflow discovery from configured directories as declarative
  extension rather than a plugin runtime
- workflow and audit report outputs for machine-readable review and archival
  flows
- stricter benchmark corpora and gates for workflow discovery, workflow
  execution, and audit query paths
- durable review runs for recurring workflow and audit loops
- review status, review diffs, and read-only remediation previews for
  supported audit findings over the composed workbench surface
- declarative workflow compatibility, review routines, report profiles, and
  workbench packs that package reusable workbench assets as data rather than
  executable plugin code
- stronger CLI and headless workflows built from that same exploratory model
- first-class everyday CLI operations for sync, search, files, nodes, refs,
  agenda, graph export, safe note creation, capture, dailies, identity, and
  metadata over the canonical daemon boundary
- stable hooks and extension points around exploration only after the first
  composed workbench has proven its shape
- richer task, ref, and citation-driven navigation

Exit criteria:

- saved artifacts can be persisted, replayed, and executed through the same
  exploration semantics already proven in the cockpit
- the first public headless surface remains task-shaped rather than becoming a
  broad CLI or extension platform story
- JSON-first machine use is real and documented without claiming broader
  platform maturity
- common research routines can be executed as named workflows rather than only
  as isolated commands
- corpus-health audits are first-class query surfaces over the canonical
  daemon boundary
- workflow discovery remains bounded and declarative rather than becoming a
  plugin runtime
- workflow and audit output surfaces are documented as CLI contracts rather
  than raw transport guidance
- scale gates cover the composed workbench paths directly
- review state is explicit, durable, and separate from ordinary notes and saved
  exploration artifacts
- users can compare repeated workflow and audit runs before deciding whether
  any remediation is appropriate
- remediation surfaces preview safe audit cleanups without implying broad
  automated mutation
- imported declarative workbench assets are explicit and separate from notes,
  review runs, and saved exploration artifacts
- workflows, review routines, report profiles, and packs can be shared and
  validated without implying a plugin runtime, MCP surface, agent adapter, or
  raw RPC wrapper layer
- pack validation, import/export, cataloging, and routine execution are
  documented as the declarative workbench extension surface, not as executable
  plugin or raw transport guidance
- everyday CLI operations cover ordinary slipbox work without bypassing
  `slipbox serve`, rebuilding semantics in the CLI, or widening into raw RPC
  wrappers
- Emacs and CLI are documented as two first-class surfaces over one source of
  truth and one daemon-owned read/write model
- structural rewrite and remediation-apply flows remain outside this milestone
  until they have a separate product surface and stability line
- users can script and automate meaningful exploratory workflows
- buffer and query surfaces compose without reaching into unstable internals
- headless surfaces reuse the exploratory model already proven in the cockpit
- any later extension surface adds capability without bending core architecture

* Milestone 5: Dependable Platform

Make the system boring in the best sense: dependable, extensible, and stable.

Outcomes:

- stable public surfaces
- documented compatibility and upgrade expectations
- clear performance envelopes
- routine release engineering

Exit criteria:

- the project can evolve without repeated rescue refactors
- packagers and users can rely on stable interfaces
- the codebase still grows by module rather than monolith

* Non-Milestones

The following are explicitly not milestones by themselves:

- adding visually impressive graph features
- increasing the raw command count
- imitating every historical detail of `org-roam`
- adding large speculative surfaces without a clear relation to the vision
